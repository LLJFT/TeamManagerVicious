# THE BOOTCAMP Multi-Game Management Platform

## Overview
The Bootcamp Multi-Game Management Platform is a comprehensive system designed for "Bootcamp" organization to centralize and streamline operations across 29 games, managing 116 teams. Its primary purpose is to provide a unified solution for team scheduling, event management, player/staff administration, and robust access control within a large-scale, multi-game esports environment. The platform offers capabilities like team chat, statistics dashboards, and seasonal management to enhance organizational efficiency and competitive performance.

## Theme System
Six themes ship with the app, registered in `client/src/hooks/use-theme.tsx` (`siteStyles`) and defined as CSS classes in `client/src/index.css`. Order matters in the picker UI:
1. **Light Pro** (`.theme-light-pro`, also mirrored to `:root`) — DEFAULT. Warm off-white surfaces (`210 24% 98%`), white cards, refined cool-gray borders (`215 16% 88%`), muted steel-blue primary (`210 64% 38%`), muted-teal accent. Tuned for dense data screens (tables, leaderboards, analytics). WCAG AA contrast for body and muted text.
2. **Default Dark** (`.theme-default-dark`) — Onyx + Crimson, the original Bootcamp brand dark.
3. **Ocean Blue** / **Ruby Red** / **Minimal Dark** / **Carbon Black** — alternate dark themes.

`ThemeProvider` (`client/src/hooks/use-theme.tsx`) defaults to `light-pro`, persists choice to `localStorage["site-style"]`, and toggles `theme-*` classes on `<html>`. Any user with a previously-saved preference (e.g. `default-dark`) keeps it; new visitors get Light Pro. The legacy `.dark` class is preserved for backwards compatibility with any third-party CSS that may target it. Status/role utility classes (`.status-*`, `.role-*`, `.event-*`, `.result-*` at the bottom of `index.css`) use light defaults + `dark:` variants and adapt automatically.

## System Architecture
The platform is built on a multi-game and multi-roster architecture, ensuring strict data isolation by `gameId` and `rosterId`. It implements session-based authentication with `bcrypt` for secure password hashing and a two-step registration approval process. A hierarchical role-based access control system (`super_admin`, `org_admin`, `game_manager`, `coach_analyst`, `player`) manages granular permissions across 39 types.

The frontend utilizes Shadcn UI components within a modern sidebar layout, supporting dark mode for a consistent user experience. Key UI elements include dynamic navigation, roster card displays, an organization dashboard, and a calendar with color-coded event previews. User and permission management are handled via a CRUD interface, and a settings page allows for organization-level configurations including dynamic theming. Loading skeletons are integrated to improve user experience.

Core features include unique display names, activity logging, a secure forgot password flow, and a dynamic event type system. Roster and player management support flexible configuration, user-player linking, and attendance tracking. Game configurations allow for per-roster side definitions, hero pools, and opponent management with logo uploads. Opponent creation is idempotent, preventing duplicates. Hero role management offers CRUD functionality with reordering and enable/disable options.

Competitive systems include a configurable Hero Ban System (HBS) with reusable presets (simple, rainbow_flexible, custom modes) and a Map Veto System (MVS) supporting ban, pick, decider, and side choice actions with editable sequences and server-enforced limits. Statistics and scoring support two-sided match stats with per-mode score configuration and multi-round scoring. A Media Library provides `super_admin` access to aggregate and manage all image URLs, supporting nested folders and batch uploads. Shared analytics filters (`AnalyticsFilterBar` + `useAnalyticsFilters` + `applyAnalyticsFilters`) are available for event type and date scoping across various statistical views like Trends, Map Insights, Hero Insights, Draft Stats, and Team Leaderboard.

The platform includes a "Load Example Data" feature for `super_admin`s to seed rosters with rich, template-driven data, supporting full template fidelity for game configurations, opponents, players, heroes, and competitive system presets. It also features Game Templates for reusable per-team configuration snapshots and a Game Slug System for unique, normalized game URLs. A robust Dataset/Seed System automatically seeds new rosters based on explicit templates or curated defaults. Draft & Scouting Analytics provide roster-scoped insights into maps, heroes, and bans, aggregating data into ranked frequency lists with win-rates and drill-down capabilities. Leaderboards (Team and Player) offer ranked statistics by opponent, map, game mode, and event type, with detailed player stats and hero specialization insights. Overlay scroll containment has been implemented across all Shadcn UI primitives to ensure proper scrolling within long content without affecting the main page. Map, Hero, and Trend Insights provide advanced statistical analysis, including win-rates by side, hero priority scores, ban efficiency, meta presence, and monthly performance trends. User experience features include in-app tutorials, timezone-aware schedule countdowns, and robust data safety. The system is designed for autoscale deployment with idempotent bootstrapping scripts.

The backend uses Node.js 20 with PostgreSQL 16.

## OCR Scoreboard Ingestion (hardened May 2026)
- Upload pipeline at `POST /api/games/:matchId/ocr-scans` enforces three gates before any persistence: (1) extension allowlist (jpg/jpeg/png/webp/bmp) via multer fileFilter, (2) magic-byte check via `hasValidMagicBytes`, (3) post-OCR `evaluateScoreboardSignal()` heuristic in `server/ocr.ts`. The third gate is **extract-what-exists / never-invent**: it only hard-rejects images that show no scoreboard signal at all (logos, posters, blank images — `reason: "ocr_no_text"` or `"no_scoreboard_signal"`). Partial / cropped / low-resolution scoreboards are accepted and flagged `validation.partial=true` so the review UI surfaces a soft amber banner explaining that blanks are intentional. Hard-rejected uploads return 422 with `{error,reason,confidence,signals,message}` and do NOT persist any scan or upload to object storage.
- `POST /api/ocr-scans/:id/confirm` now defaults to **merge** mode: only inserts heroes/participants/stats whose `(side, player, …)` tuple isn't already present, preserving existing rows; re-confirming the same scan or re-uploading the same match is idempotent. Pass `{overwrite:true}` for legacy wipe-and-replace. Game scores are written only when missing OR `overwrite=true`. Response shape: `{ok, mode, scan, counts, skipped}` (counts and skipped both include `opponentStats`).
- `PATCH /api/ocr-scans/:id` zod-validates the `editedCandidate` payload via `ocrParsedCandidateSchema` from `shared/schema.ts`.
- `DELETE /api/ocr-scans/:id` marks a scan as `discarded` (image kept for audit) — surfaced as a "Discard scan" button in `client/src/pages/OcrScanReview.tsx`.
- Review UI shows a per-row OCR + confidence badge (High/Med/Low based on fuzzy-match avg), a scan-level scoreboard-validation badge, a low-confidence banner when any row is below 50%, and an explicit "Replace existing" toggle that switches the confirm button to a destructive-styled "Replace & import".

### Extract-what-exists / never-invent (May 2026)
- The whole pipeline preserves only what is actually visible in the image. Missing values stay `null`/empty — they are never guessed, auto-filled, or hallucinated. This applies to player names, opponent names, hero names, every stat field (kills/deaths/assists/final hits/damage/damage-blocked/healing/etc), map, mode, side, and score.
- `parseScoreboardText()` walks the OCR token grid and assigns numeric tokens to stat fields by positional order; if fewer numeric tokens are present than stat fields, only the visible ones are filled (`stats[sf.id]` is left unset for the rest). Rows with no name/hero match but ≥2 numeric tokens are still kept (side="unknown") so the coach can assign them in review.
- `evaluateScoreboardSignal()` no longer rejects partial scoreboards. Only zero-signal images (logo / poster / blank) trip the 422; everything else is saved with `partial: confidence < 0.55 || rowCount < 2 || matchedPlayers+matchedOpponents < 2 || matchedHeroes < 2`.
- `POST /api/ocr-scans/:id/confirm` skips empty stat values (`if (value === "" || value === null || value === undefined) continue`) so blank fields never produce ghost rows in `player_game_stats` / `opponent_player_game_stats`.
- The review UI shows an amber **Partial extraction** banner whenever `validation.partial` is true. Empty stat inputs render as blank text boxes — the coach fills in what the system couldn't see.

## Load Example Data — role-based stat seeding (May 2026)
- `server/roster-reset.ts` `loadExampleData()` now generates **role-aware** stat values for every seeded `player_game_stats` and `opponent_player_game_stats` row. The new `normalizeRole()` maps both our roster vocabulary (Tank/DPS/Support/Flex) and Marvel Rivals template vocabulary (Vanguard/Duelist/Strategist) to a 4-bucket role; `roleStatRange(role, statFieldName)` then returns a `[min, max]` range matched by case-insensitive substring of the stat field name. Order matters (most specific first): `block` → tank-heavy Damage Blocked, `heal` → support-heavy Healing, `death` → role-flat, `damage|dmg` → DPS-heavy, `assist` → support/tank-heavy, `kill|final|elim` → DPS-heavy. Generic numeric fallback returns `[0, 12]`.
- Opponent fallback rosters bumped from 5 to 6 players (`FALLBACK_OPP_ROLES` = 2 Tank / 2 DPS / 2 Support) so every Marvel Rivals opposing side is fully populated for analytics. The opponent-player insert loop now uses `FALLBACK_OPP_ROLES.length` instead of the hard-coded `5`. The `oppPlayerRoleById` map is built alongside `opponentPlayerIdsByOpp` so the opp-stat insert can look up each opp player's role without an extra DB roundtrip.
- The seed return shape now includes `playerStatsRows` and `opponentStatsRows`. The Dashboard "Example data loaded" toast (`client/src/pages/Dashboard.tsx`) was rewritten to surface those counts plus heroes, hero picks, opponents, and opp-players.
- Team Comps is **derived** from `game_heroes` (no separate table) — varying hero picks per game per player already produces multiple comp archetypes. Hero picks per game still use `pickPool(side_size)` which shuffles the hero pool.

## Event Result Settle Timer (May 2026)
- New `events.result_source` (`'pending' | 'auto' | 'manual'`, default `'pending'`) and `events.last_game_change_at` (`timestamp`) columns added in `shared/schema.ts`. Idempotent migration in `server/migrations/add-event-result-source.ts` runs from the deferred boot chain in `server/index.ts` (after dedupe + base64-migration) — adds columns with `IF NOT EXISTS`, then backfills existing events whose `result IN ('win','loss','draw')` to `result_source='auto'` and the rest to `'pending'`.
- `server/storage.ts`:
  - `addGame` / `updateGame` / `removeGame` no longer call `recomputeEventResult` synchronously. They now call `bumpEventLastGameChange(eventId)` which sets `last_game_change_at = NOW()` and flips `result_source` to `'auto'` (only when not already `'manual'`). The actual recompute is deferred so the operator's chain of edits doesn't keep flipping the result.
  - `recomputeEventResult(eventId, opts?)` now (a) accepts an explicit `teamId` so the scheduler can call it without a request context, and (b) honors `respectManual` (default true) by reading `events.result_source` first and skipping any update when it is `'manual'`. On success it stamps `result_source='auto'`.
  - `runDueEventResultAutoComputes()` is the new scheduler entry point. Selects events where `last_game_change_at IS NOT NULL AND last_game_change_at < NOW() - INTERVAL '5 minutes' AND result_source IN ('pending','auto')` (limit 500/tick), recomputes each, then nulls `last_game_change_at` so the same event isn't reprocessed every 60s.
- `server/event-result-scheduler.ts` runs `setInterval(tick, 60_000)` plus a `setTimeout(tick, 5000)` first-pass to catch events whose window elapsed during downtime. Started from `server/index.ts` after the migration runs.
- `PUT /api/events/:id` (`server/routes.ts`) **no longer strips `result`**. When the client sends a `result` that differs from the existing value, the route flips `resultSource='manual'`. A no-op resend is dropped so the source doesn't oscillate.
- `client/src/pages/EventDetails.tsx`:
  - The result `<Select>` is no longer disabled — `eventResult` is the controlled value, and `handleSaveEventDetails` now sends `result` as part of the PUT payload.
  - A `<Badge>` shows the source: "Manually set" (default variant), "Auto-computed" (secondary), or "Pending" (outline).
  - The hint text below the dropdown:
    - For `manual`: "Manually set — change the result above to override or pick a different value to update. The 5-minute auto-compute will not touch this event."
    - For `pending` / `auto` with a recent `lastGameChangeAt`: "Will auto-set in ~Xm SSs from the games below. Pick a value above to override."
    - Otherwise: "Auto-computed from the games below after a 5-minute settle window. Pick a value above to override."
- Manual-override semantics: choosing any non-pending value via the dropdown locks the event. To return to auto, change a game result (which bumps `last_game_change_at` but preserves `manual`) — manual stays manual until the user picks `pending` or another value matching the auto-derived result. Editing a game while the event is manual will still bump `last_game_change_at` but the scheduler skips manual-source events.

### Extract-first / map-later (May 2026 follow-up)
- The parser in `server/ocr.ts` no longer guesses team membership when the IGN can't be matched. The old "split rows in half so the reviewer sees both sides populated" heuristic was the root cause of `Player Leaderboard → Our players` showing nothing while opponents appeared: half of our matched rows were re-labelled `side="opponent"` then silently dropped on confirm because they had no `matchedOpponentPlayerId`. Unmatched rows are now emitted with `side: "unknown"` (`OcrPlayerRow.side` accepts `"us" | "opponent" | "unknown"`).
- `POST /api/ocr-scans/:id/confirm` rejects with HTTP 400 `{ error: "unassigned_rows", unassignedRowIdx: number[] }` if any row is still `unknown`, or has a side but no matching player id. No partial imports, no silent drops.
- The confirm route now writes opponent stats to `opponent_player_game_stats` symmetrically with our-side stats (`saveOpponentPlayerGameStats(matchId, rows, gameId)`); merge dedup key is `(opponentPlayerId, statFieldId)`. Both branches also dedup intra-import duplicates by `(side, owner)` and `(owner, statFieldId)` before persistence so manual re-mapping in review can't insert duplicate rows.
- Review UI (`client/src/pages/OcrScanReview.tsx`) renders three sections: **Needs assignment** (any `unknown` row), **Our team**, and **Opponent**. Each row has an explicit Side selector (Our team / Opponent / Unknown); flipping side clears the conflicting matched id. The confirm button is disabled with an "Assign N rows first" label until every row is assigned, and a destructive banner explains that captured numeric stats are preserved while the coach maps players. On success the mutation invalidates `/api/player-game-stats`, `/api/opponent-player-stats`, `/api/match-participants`, and `/api/game-heroes` so leaderboards refresh immediately.
