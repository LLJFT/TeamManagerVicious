# Vicious Esports Multi-Game Management Platform

## Overview
This project is a comprehensive multi-game esports management platform designed for "The Vicious" organization. It centralizes and streamlines operations for 29 games, each supporting four rosters, totaling 116 teams. Key capabilities include team scheduling, event management, player/staff administration, robust role-based access control with a two-step registration approval, team chat, statistics dashboards, and seasonal management. The platform aims to provide a unified solution for managing a large-scale, multi-game esports entity.

## User Preferences
Preferred communication style: Simple, everyday language.

## System Architecture

### Core Architecture
The platform is built to support a multi-game and multi-roster structure, isolating data strictly by `gameId` and `rosterId`. It uses session-based authentication with `bcrypt` for password hashing and a two-step approval process for registration. A hierarchical role-based access control system manages permissions, including `super_admin`, `org_admin`, `game_manager`, `coach_analyst`, and `player` roles, with granular control over 39 permission types.

### Frontend and UI/UX
The frontend features a modern sidebar layout with dark mode and utilizes Shadcn UI components for a consistent user experience. It includes dynamic navigation, a roster card display on the home page, an organization dashboard, and a calendar with color-coded event previews. User role and permission management are handled via a CRUD interface, and a settings page allows for organization-level configurations including dynamic theming. Loading skeletons are used to enhance user experience.

### Key Features
-   **Data Management**: Unique display names, activity logging, secure forgot password flow, and a dynamic event type system.
-   **Roster & Player Management**: Roster configuration, user-player linking, and attendance tracking for events.
-   **Game Configuration**: Per-roster configurable side definitions, hero pools, opponent management with logo uploads. Supports a per-team single-mode flag to simplify game mode presentation.
-   **Image Persistence**: All image uploads are handled via Replit Object Storage, generating short, stable URLs to avoid payload size issues.
-   **Opponent Idempotency**: Opponent creation is idempotent, preventing duplicate entries. Opponent cards display team logos and provide a "View Roster" button.
-   **Hero Role Management**: Per-game role definitions are managed with CRUD functionality, reordering, and enable/disable options. Renaming roles cascades updates, and deletion requires hero reassignment.
-   **Competitive Systems**:
    -   **Hero Ban System (HBS)**: Configurable, reusable per-roster presets with support for simple, rainbow_flexible, and custom modes, with editable action sequences and server-enforced limits.
    -   **Map Veto System (MVS)**: Configurable, reusable per-roster presets supporting ban, pick, decider, and side choice actions, with editable sequences and server-enforced limits.
-   **Statistics & Scoring**: Two-sided match stats with per-mode score configuration, multi-round scoring, and detailed player statistics.
-   **Media Library**: A super_admin feature aggregating all image URLs across the platform, including custom folders. Supports nested folders, search, and image uploads.
-   **Game Templates**: Reusable per-team configuration snapshots for games, including modes, maps, heroes, stats, and competitive systems. Applying a template is a destructive operation that wipes and re-initializes roster-scoped data.
-   **Game Slug System**: Each `supportedGames` row has a unique `slug` (text, unique index). The slug is the first segment of every roster URL: `/<game-slug>/<roster-code>` where `<roster-code>` is the 10-digit numeric `rosters.code` (NOT a UUID). Server-side normalizer `normalizeSlug()` (server/routes.ts) lowercases, strips non `[a-z0-9-]`, collapses repeated dashes, trims edges, and rejects empty results. `POST /api/supported-games` and `PUT /api/supported-games/:id` validate + pre-check uniqueness (returning 409 with a friendly message on conflict, 400 on invalid). Adding a new game also creates 4 default rosters (Team 1–4) and auto-seeds each via `storage.seedNewRoster` — failures are logged per-roster but never roll back game creation (returns `seedSummary` array). The Settings → Game Management UI shows a live `URL preview: /<slug>/...` with a red conflict warning + disabled Add Game button when the slug is invalid or duplicate.
-   **Dataset / Seed System**: Every newly created roster (`POST /api/rosters`, `POST /api/supported-games/:gameId/rosters`) is auto-seeded via `storage.seedNewRoster`. Precedence: explicit `templateId` → `findDefaultTemplateForGame` (most-recently-updated team+game template) → curated per-game defaults from `server/defaults/gameDefaults.ts` (Valorant, Overwatch, Marvel Rivals, LOL, CS2, generic fallback). Default path seeds roster roles, sides, event categories+sub types, game modes/maps/stat fields, heroes (Overwatch & Marvel Rivals only), hero ban + map veto presets, ≥2 real opponents from `OPPONENT_SEEDS_BY_GAME_SLUG` each with N generic players (IGN-baked, role round-robin). `force=false` short-circuits with `source:"skipped"` if any roster-scoped table already has data. `force=true` runs the same FK-safe wipe (nulls `users.playerId`, deletes children before parents) used by `applyGameTemplate`. Auto-seed failures are caught and never roll back roster creation. Manual super_admin re-seed: `POST /api/admin/rosters/:id/seed` (reset password gated) + Seed Roster card in the Reset Roster tab of the roster Dashboard.
-   **Draft & Scouting Analytics**: Roster-scoped Draft Stats page (Maps + Heroes tabs) aggregates `gameMapVetoRows`, `gameHeroBanActions`, and `gameHeroes` into ranked frequency lists with optional opponent filter, win-rate per map/hero, and drill-down popovers linking to source events. Opponent-specific Scouting Insights are surfaced inside each expanded opponent card on the Opponents page. Three new GET endpoints (`/api/hero-ban-actions`, `/api/map-veto-rows`, `/api/game-heroes`) are protected by `requireGameAccess` and auto-scoped via `GAME_SCOPED_PREFIXES`. Hero-play counts dedupe per (matchId, heroId, playerId/opponentPlayerId) to avoid multi-round inflation.
-   **Leaderboards**: Two roster-scoped pages under the Statistics sidebar group, both gated by the `view_statistics` permission and following the DraftStats convention (all hooks declared before any early return, opponent filter dropdown, `AccessDenied`/`StatsSkeleton` guards, drill-down popovers to source events). **Team Leaderboard** (`/team-leaderboard`) ranks roster records (W-L-D + win-rate) by opponent, map, game mode, and event type — aggregated client-side from existing `/api/games`, `/api/maps`, `/api/game-modes`, `/api/opponents`, `/api/events`. **Player Leaderboard** (`/player-leaderboard`) shows three tabs: Attendance (matches played per player from `match_participants` where `side="us" && played`), Stat Leaders (per stat-field totals + averages from `player_game_stats`), and Hero Specialization ((player, hero) pairings from `gameHeroes` ranked by picks with pairing win-rate). Two new GET endpoints (`/api/match-participants`, `/api/player-game-stats`) are protected by `requireGameAccess` + `requirePermission("view_statistics")` and auto-scoped via `GAME_SCOPED_PREFIXES`. All aggregations dedupe by their natural composite keys to avoid multi-round inflation.
-   **Map / Hero / Trend Insights**: Three additional roster-scoped Statistics pages, all gated by `view_statistics` and following the DraftStats convention. **Map Insights** (`/map-insights`): Overview (best/worst with min-3-game qualifier + ranked WR), Side WR (per-map per-side round win-rate from `gameRounds.sideId` + `teamScore`/`opponentScore`), Trend (monthly WR per map matrix), and vs Opponent (map × opponent WR matrix). **Hero Insights** (`/hero-insights`): Priority Score (pick% + ban% combined per hero across all draft phases), Never Played + Always Banned lists, Ban Efficiency (our bans → opponent's WR on the banned hero; their bans → our WR on the banned hero), Meta Presence (% of scoped matches in which a hero appears anywhere), and Hero Pool by Player (unique-hero count + most-played + best/worst by WR with min-2 qualifier). **Trends** (`/trends`): summary cards (games, WR, current/longest streaks), monthly WR (recharts LineChart) + stacked bar (W/L/D), Best/Worst monthly periods, Avg Score (parses `"X-Y"` from `games.score` with `±`-margin extremes), and By Side (overall round WR per side). One new GET endpoint (`/api/game-rounds`) is added — protected by `requireGameAccess` + `requirePermission("view_statistics")`, auto-scoped via `GAME_SCOPED_PREFIXES`, optional `opponentId` filter joined through `games`. All page hooks are declared before any `AccessDenied`/`StatsSkeleton` early return, picks are deduped per `(matchId, heroId, playerId|opponentPlayerId)`, and a streak walks the chronologically-sorted event list.
-   **User Experience**: In-app tutorial, schedule countdowns with timezone awareness, and robust data safety with no auto-deletion.
-   **Deployment & Development**: Designed for autoscale deployment, with startup scripts for idempotent bootstrapping.

### Database and Deployment
The platform uses PostgreSQL for its database, leveraging extensive game-scoped and roster-scoped tables. It is designed for deployment on platforms like Replit or Vercel/Neon, using Node.js 20 with PostgreSQL 16.
