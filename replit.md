# Vicious Esports Multi-Game Management Platform

## Overview
This project is a comprehensive multi-game esports management platform designed for "The Vicious" organization. It centralizes and streamlines operations for 29 games, each supporting four rosters, totaling 116 teams. Key capabilities include team scheduling, event management, player/staff administration, robust role-based access control with a two-step registration approval, team chat, statistics dashboards, and seasonal management. The platform aims to provide a unified solution for managing a large-scale, multi-game esports entity.

## User Preferences
Preferred communication style: Simple, everyday language.

## System Architecture

### Core Architecture
The platform is built to support a multi-game and multi-roster structure, isolating data strictly by `gameId` and `rosterId`. It uses session-based authentication with `bcrypt` for password hashing and a two-step approval process for registration. A hierarchical role-based access control system manages permissions, including `super_admin`, `org_admin`, `game_manager`, `coach_analyst`, and `player` roles, with granular control over 39 permission types.

### Frontend and UI/UX
The frontend features a modern sidebar layout with dark mode and utilizes Shadcn UI components for a consistent user experience. It includes dynamic navigation, a roster card display on the home page, an organization dashboard, and a calendar with color-coded event previews. User role and permission management are handled via a CRUD interface, and a settings page allows for organization-level configurations including dynamic theming. Loading skeletons are used to enhance user experience.

### Key Features
- **Data Management**: Unique display names, activity logging, secure forgot password flow, and a dynamic event type system.
- **Roster & Player Management**: Roster configuration, user-player linking, and attendance tracking for events.
- **Game Configuration**: Per-roster configurable side definitions, hero pools (with default seeding for Marvel Rivals), and opponent management with team logo uploads. Each game also supports a per-team **single-mode** flag (settings key `single_mode_game`, scoped by teamId+gameId) that hides the game-mode dimension entirely — Maps and Stat Fields render as a single flat list, the new-game form skips the mode picker, and existing per-mode data (modes, maps, stat fields, score configs) is preserved so the toggle is fully reversible. Toggle lives on the Dashboard → Game Config tab via "Remove" / "Add Game Mode" buttons (gated by `manage_game_config`). API: `GET/PUT /api/game-config/single-mode?gameId=X`. Score Configuration card and Heroes role groups on the Dashboard are now collapsed by default to reduce visual noise. Opponent logos appear next to "vs <name>" everywhere opponents are shown (event lists, history, opponent stats, event details, player stats, upcoming countdowns, event-dialog picker), via the shared `OpponentAvatar` component (falls back to initials when no logo is set).
- **Image Persistence**: All image uploads (opponent logos, hero images, map images, event scoreboards, media-library entries) go through `POST /api/objects/upload`, which uploads to Replit Object Storage and returns a short, stable `/objects/uploads/<id>.<ext>` URL served by `GET /objects/uploads/:filename` (auth-required, `nosniff`, attachment fallback for non-image MIME). If object storage is unreachable for an image upload the route returns a clean 503 — we deliberately no longer emit base64 data URLs for images, because previously those would get persisted into JSONB / text columns and balloon template payloads past the 10 MB Express body limit (causing 413s and editor lag). Non-image uploads still fall back to local disk in development. The `OpponentAvatar` and image-cell components handle the resulting URL forms transparently. The `ObjectUploader` component supports `multiple` mode for batch uploads (sequential per-file, per-file progress, per-file `onUploaded` callback that includes the original `File` so callers can preserve filenames). On boot, `migrateBase64Images` (in `server/migrations/migrate-base64-images.ts`) scans `heroes.image_url`, `maps.image_url`, `opponents.logo_url`, `games.image_url`, `media_items.url`, and the `game_templates.config` JSONB tree (`heroes[].imageUrl`, `maps[].imageUrl`, `opponents[].logoUrl`) for any legacy `data:image/...;base64,...` value and re-uploads it to object storage, writing the new short URL back. Every UPDATE is conditional on the original value still being the same data URL, so concurrent user edits during the migration are preserved; failures and conflicts leave the original value untouched and are logged for the next boot to retry.
- **Opponent Idempotency**: `POST /api/opponents` is now get-or-create at the storage layer, deduping on `(teamId, gameId, rosterId, lower(name))`. Lazy-creation flows from Analytics → Opponents (e.g. when only event-text references an opponent) cannot produce duplicate rows even under concurrent clicks.
- **Analytics → Opponents**: Each opponent card shows the team logo (via `OpponentAvatar`) and a "View Roster" button gated by `manage_game_config`. Clicking opens the shared `OpponentRosterDialog` (extracted to `client/src/components/OpponentRosterDialog.tsx`), lazy-creating an opponents row by name if one doesn't yet exist — never deletes opponents, never wipes roster players, never breaks event/history links.
- **Hero Role Management**: Per-game role definitions managed via the "Manage Roles" button inside Heroes Configuration (gated by `manage_game_config`). Roles are stored in `hero_role_configs` (teamId + gameId + name unique). Supports CRUD, reorder, enable/disable. Add/Edit Hero dialogs pull options from this list. Renaming a role atomically cascades the new name to all heroes using it. Deleting a role-in-use returns 409 with `heroesAffected` and requires `?reassignTo=<roleConfigId>` to migrate heroes; deletion runs in a transaction. The "All" tab in Heroes Configuration is a filter only; "Other" only appears when heroes truly have unmapped roles.
- **Competitive Systems**:
    - **Hero Ban System (HBS)**: Configurable, reusable per-roster presets with support for simple, rainbow_flexible, and custom modes. Each game can opt-in to a preset, with an editable sequence of actions (ban/protect) and server-enforced limits.
    - **Map Veto System (MVS)**: Configurable, reusable per-roster presets supporting ban, pick, decider, and side choice actions. Each game can opt-in, with an editable sequence of actions and server-enforced limits. Map images are supported for visual representation.
- **Statistics & Scoring**: Two-sided match stats (replacing legacy player stats) with per-mode score configuration, multi-round scoring, and detailed player statistics breakdowns. Opponent event history and share functionality are also included.
- **Media Library** (super_admin): Top-level page at `/media-library` plus a "Browse Library" launcher inside Game Templates → Maps tab. Aggregates every image URL stored on the platform — `maps.image_url`, `heroes.image_url`, `opponents.logo_url`, and `games.image_url` (per-game scoreboard uploads) — plus user-defined custom folders (`media_folders` + `media_items`, team-scoped, super-admin only). The `media_folders` table supports nested subfolders via `parent_id` and a `game_id` column that scopes a folder (and all of its descendants) to a specific game group. Returned by `GET /api/media-library` as `{ games: MediaGame[], customFolders: CustomFolder[] }` (super_admin-gated, scoped to the current team). Page is organized by built-in game cards (collapsible, default collapsed) and a "Custom Folders" group; each game group also renders its own custom subfolders alongside the built-in image categories. Each tile shows a thumbnail, name, URL, and a copy button; clicking the thumbnail copies the URL. Searchable across both games and custom folders. The shared `MediaLibraryBrowser` component supports an `onSelect(url)` callback so the Game Templates → Maps tab can open it as a dialog (filtered to the template's game; custom folders auto-hidden in that dialog) and inject the chosen URL directly into the row. Custom folder management (super-admin only): `POST /api/media-folders {name,sortOrder,parentId?,gameId?}` — when `parentId` is set the new folder inherits the parent's `gameId`, so a subfolder can never escape its tree; `PUT /api/media-folders/:id` strips direct `gameId` rewrites and validates any new `parentId` against same-team + not-self + not-a-descendant (cycle prevention); `DELETE /api/media-folders/:id` recursively purges every descendant folder (a `visited` set guards against runaway loops if legacy rows ever form a cycle); `POST /api/media-items {folderId,name,url,sortOrder}`, `PUT/DELETE /api/media-items/:id`. The page exposes nested "New Folder" buttons (root, sub, and per-game) plus per-folder Upload (uses the same `ObjectUploader` → `/api/objects/upload` pipeline), with confirmation dialog before folder delete. The recursive tree builder in `GET /api/media-library` carries a `visited` set per branch so cyclic legacy data can never hang the response. Library used here for grouping arbitrary org logos / brand assets like "ORG > T1 ORGS".
- **Game Templates** (super_admin): Reusable per-team configuration snapshots stored in `game_templates` (id varchar UUID, teamId, gameId, unique name+code per team, JSONB `config` containing gameModes/maps/heroes/statFields/scoreConfigs/eventCategories/eventSubTypes/availabilitySlots/opponents/sides/rosterRoles/heroRoles/heroBanSystems/mapVetoSystems + singleModeGame). Pages: `/game-templates` (list with create/copy-code/delete) and `/game-templates/:id` (tabbed editor mirroring the Dashboard config tabs minus Users/Roles). The editor exposes 14 tabs: General, Game Modes, Maps, Heroes, Hero Roles, Hero Ban, Map Veto, Stat Fields, Score Config, Sides, Event Categories, Sub Types, Availability, Roster Roles, Opponents. Codes use the `<gameAbbrev3>-<6 alphanumeric>` format. CRUD endpoints (`/api/game-templates`) are super_admin-only; `GET /api/game-templates/by-code/:code` and `POST /api/game-templates/apply` (body `{templateId|code}`, query `gameId`+`rosterId`) require `manage_game_config` AND pass `verifyObjectScope` so callers can only apply to rosters they actually have access to. Apply runs in a single DB transaction: deletes roster-scoped (teamId+gameId+rosterId) game_modes, maps, stat_fields, heroes, event_categories, event_sub_types (deleted before categories due to FK cascade), availability_slots, opponents, sides, roster_roles, hero_ban_systems, map_veto_systems, then re-inserts from the template config with tempId→UUID remapping (so foreign-key fields like `gameModeId` and event sub-type → category links survive). **Hero Roles are template-scoped only** — applying a template never writes to the team's `hero_role_configs` table; the template's `heroRoles` list is consumed only inside the editor (it drives the Heroes tab role `<Select>`, falling back to `["Other"]` when empty) so two templates on the same team can advertise different hero-role sets without clobbering each other. Null-roster shared rows are NEVER touched. Players, games, results, events, attendance, and history are NEVER touched. The Dashboard "Reset Roster" tab includes an "Apply Game Template" card with code lookup, preview tally, and a confirmation dialog.
- **User Experience**: In-app tutorial for onboarding, schedule countdowns that account for event timezones, and robust data safety measures with no auto-deletion.
- **Deployment & Development**: Designed for autoscale deployment, with startup scripts for idempotent bootstrapping of default data, migrations, and directory creation.

### Database and Deployment
The platform uses PostgreSQL for its database, leveraging extensive game-scoped and roster-scoped tables. It is designed for deployment on platforms like Replit or Vercel/Neon, using `npm run build` for frontend and server bundling, and `npm run start` for the Node.js production server. The environment is Node.js 20 with PostgreSQL 16.
