OVERVIEW
═══════════════════════════════════════
Add a "Game Templates" section in the Admin Home page.
Templates are NEVER auto-created, auto-applied, or auto-linked.
Visible and accessible to Super Admin only.
A template is a full manual copy of a roster Dashboard config.
Apply it to any roster (same game) using a fixed code.

═══════════════════════════════════════
STEP 1 — Game Templates page (Super Admin only)
═══════════════════════════════════════
Add new top-level page: "Game Templates"
- Visible ONLY to Super Admin
- Add to Admin Home navigation, hidden from all other roles

Each template card shows:
- Template name
- Game it belongs to
- Fixed immutable code (e.g. "OW-X7K2P1") with one-click copy
- Created date
- Edit → opens template editor
- Delete → with confirmation

"+ New Template" button:
- Asks for: Template Name + Game (dropdown of existing games)
- Auto-generates immutable code on creation
- Opens template editor immediately after creation

═══════════════════════════════════════
STEP 2 — Template editor (identical to roster Dashboard)
═══════════════════════════════════════
Template editor = exact same UI as roster Dashboard.
Include ALL these tabs:
- Game Modes (respects single_mode_game flag)
- Maps + map images
- Heroes + hero images + roles
- Stat Fields
- Score Configuration
- Event Categories
- Availability Times
- Opponents

REMOVE only:
- Users tab
- Roles tab

Everything works identically to the roster Dashboard.
Images uploadable and stored same way (object storage or base64).
Templates stored in isolated data scope.
Editing a template NEVER affects any live roster.
Templates NEVER auto-populated — admin fills manually.

═══════════════════════════════════════
STEP 3 — Apply template to a roster
═══════════════════════════════════════
In roster Dashboard → Settings (or Reset Roster area),
add "Apply Game Template" — visible to Super Admin only.

Flow:
1. Text input: "Enter template code"
2. Validate code exists AND template game matches roster game.
   If mismatch → error: "This template is for [Game].
   This roster uses [Other Game]. Use a template for the correct game."
3. Confirmation dialog:
   "This will REPLACE:
   • Heroes and hero images
   • Maps and map images
   • Game Modes
   • Stat Fields
   • Score Configuration
   • Event Categories
   • Availability Times
   • Opponents

   Will NOT be touched:
   • Players
   • Game records and results
   • Events and history

   This cannot be undone. Continue?"

4. On confirm — single database transaction:
   a. DELETE existing config for this roster only:
      heroes, maps, game_modes, stat_fields, score_config,
      event_categories, availability_times, opponents
   b. INSERT deep copies of all template rows into roster
   c. Re-map ALL internal IDs:
      - game_mode rows → new IDs
      - maps → reference new game_mode IDs
      - stat_fields → reference new game_mode IDs
      - score_config → reference new game_mode IDs
   d. Images:
      - Object storage URL → copy string as-is
      - base64 → copy string as-is
      - All images work immediately after apply
   e. Set roster single_mode_game = template single_mode_game
   f. ANY failure → full rollback, roster untouched

═══════════════════════════════════════
STEP 4 — Schema
═══════════════════════════════════════
game_templates (
  id         serial primary key,
  name       text not null,
  code       text unique not null,  -- immutable after creation
  game_id    integer references games(id),
  team_id    integer references teams(id),
  config     jsonb not null,
  created_at timestamp default now()
)

config JSONB stores:
heroes, maps, game_modes, stat_fields, score_config,
event_categories, availability_times, opponents,
single_mode_game flag, all image data.

Code format: game abbreviation + random alphanumeric (e.g. OW-X7K2P1)
Generated once on INSERT, never updated.

If JSONB is not feasible → use child tables.
Make the decision yourself, pick the cleaner approach, implement fully.
